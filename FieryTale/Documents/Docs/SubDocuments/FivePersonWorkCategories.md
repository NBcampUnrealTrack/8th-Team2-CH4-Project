# Five Person Work Categories

## 목적

이 문서는 FieryTale 4주 프로토타입을 5명의 작업자에게 분배하기 위한 큰 작업 분류를 정리한다.

현재 기준은 주니어급 프로그래머 5명이다. 각자의 세부 역량은 아직 확정되지 않았으므로, 이 문서는 개인 이름별 배정보다 기능 영역별 책임을 먼저 나누는 기준으로 사용한다.

## 분류 원칙

5명에게 작업을 나눌 때는 다음 기준을 지킨다.

- 한 사람에게 네트워크, GAS, UI를 동시에 몰지 않는다.
- 각 작업자는 명확한 산출물과 완료 기준을 가진다.
- 1주차에는 콘텐츠 확장보다 기술 리스크 검증을 우선한다.
- 2주차에는 로비에서 매치 시작, 전투, 구조물 파괴, 넥서스 승패까지 연결한다.
- 3~4주차에는 새 시스템보다 조작감, 피드백, UI, 안정성 폴리싱을 우선한다.
- Listen Server 기준 2 vs 2 시연을 필수 목표로 두고, Dedicated Server는 선택 목표로 둔다.

## 5개 작업 분류

| 작업자 | 큰 분류 | 주요 책임 |
|---|---|---|
| A | 네트워크 / 매치 흐름 | `GameMode`, `GameState`, 팀 배정, 접속, 스폰, 매치 시작/종료, 승패 판정 |
| B | 캐릭터 / TPS 조작 | `BaseCharacter`, 이동, 카메라, 조준, 입력, 사망/리스폰 상태 연결 |
| C | GAS / 전투 시스템 | `AbilitySystemComponent`, `AttributeSet`, 기본 공격, 스킬 슬롯, Damage Effect, GameplayTag |
| D | 전장 / 구조물 / 목표물 | 2라인 블록아웃, `TowerBase`, 방어타워, 넥서스, 구조물 피해/파괴, 넥서스 취약화 |
| E | UI / 로비 / 데이터 / QA | 기본 로비, 캐릭터 선택 Data Table, HUD, 승패 메시지, 테스트 체크리스트, 3~4주차 채팅/리더보드 |

## A. 네트워크 / 매치 흐름

### 목적

서버 권한 기준으로 한 판의 시작, 진행, 종료 흐름을 관리한다.

이 영역은 FieryTale MVP의 가장 중요한 통합 축이다. 다른 작업자가 만든 캐릭터, 전투, 구조물, UI는 결국 A의 매치 상태 흐름에 연결되어야 한다.

### 주요 담당

- 프로젝트 기본 실행 흐름 확인
- Listen Server 접속 흐름
- `GameMode` 서버 전용 규칙
- `GameState` 복제 상태
- `PlayerState` 팀, 선택 캐릭터, 킬/데스 상태
- 플레이어 접속 처리
- Team A/B 자동 배정
- 팀별 스폰 호출
- 매치 상태 전이
  - `Lobby`
  - `Starting`
  - `InProgress`
  - `Ending`
  - `Finished`
- 넥서스 파괴 시 승패 확정
- 매치 종료 후 입력 제한과 결과 상태 복제

### 1주차 완료 기준

- 2명 이상 클라이언트가 같은 서버에 접속한다.
- 서버가 플레이어를 Team A/B로 배정한다.
- 각 클라이언트가 자기 팀을 확인할 수 있다.
- 팀별 스폰 위치가 분리된다.
- 넥서스 또는 테스트 구조물 파괴 시 승패 로그 또는 간단 UI가 출력된다.

### 주요 Trello 카드

- `[W1] Unreal 프로젝트 기본 구조 확인`
- `[W1] 멀티플레이 이동 복제 확인`
- `[W1] GameMode 팀 배정 구현`
- `[W1] 팀별 PlayerStart 스폰 구현`
- `[W2] 매치 시작 흐름 연결`
- `[W2] 넥서스 파괴 승패 판정`
- `[BG] Dedicated Server 빌드 생성`
- `[BG] Dedicated Server 클라이언트 접속 확인`

### 주의점

- 클라이언트가 체력, 사망, 구조물 파괴, 승패를 직접 확정하지 않게 한다.
- `GameMode`에는 서버 전용 규칙을 둔다.
- 클라이언트가 알아야 하는 값은 `GameState` 또는 `PlayerState`로 복제한다.
- Dedicated Server 작업은 Listen Server 2 vs 2 흐름이 안정된 뒤 진행한다.

## B. 캐릭터 / TPS 조작

### 목적

플레이어가 직접 조작하는 `BaseCharacter`의 이동, 카메라, 조준, 입력 감각을 만든다.

FieryTale은 TPS 시점으로 확정되었으므로, 이 영역은 게임의 체감 품질에 직접 영향을 준다.

### 주요 담당

- `BaseCharacter` 또는 공통 Pawn 구조
- 이동 입력
- TPS 카메라
- 카메라 회전
- 조준 방향 계산
- 기본 공격 입력 바인딩
- 공격 방향과 카메라 전방 연결
- 사망 상태에서 입력 차단
- 리스폰 후 조작 복구
- 멀티플레이 환경에서 이동 복제 확인

### 1주차 완료 기준

- 테스트맵에서 캐릭터가 이동한다.
- 카메라 회전과 조준 방향이 동작한다.
- 기본 공격 입력이 GAS 또는 전투 시스템으로 전달된다.
- 서버/클라이언트에서 서로의 캐릭터 위치가 보인다.
- 사망 상태에서 입력과 공격이 막힌다.
- 리스폰 후 카메라와 입력이 정상 복구된다.

### 주요 Trello 카드

- `[W1] BaseCharacter 이동/카메라 입력 연결`
- `[W1] 멀티플레이 이동 복제 확인`
- `[W1] 기본 공격 입력 바인딩`
- `[W1] 사망 상태 입력 차단`
- `[W2] 리스폰 후 캐릭터 조작 복구`
- `[W4] 조작감 개선`

### 주의점

- 1~2주 MVP에서는 과한 카메라 연출을 넣지 않는다.
- 숄더뷰처럼 너무 가까운 카메라는 피하고, 캐릭터 뒤쪽 위 TPS 기준을 유지한다.
- 정교한 클라이언트 예측이나 조준 보정은 MVP 필수 범위에서 제외한다.
- 전투 수치보다 입력, 카메라, 상태 전이 안정성을 우선한다.

## C. GAS / 전투 시스템

### 목적

GAS를 기반으로 캐릭터의 체력, 피해, 기본 공격, 스킬 슬롯, 쿨다운, 상태 태그를 구현한다.

이 영역은 캐릭터 확장성과 전투 판정의 중심이다.

### 주요 담당

- `AbilitySystemComponent` 연결
- 공통 `AttributeSet`
  - `Health`
  - `MaxHealth`
- 기본 공격 `GameplayAbility`
- 공격 스킬 슬롯
- 이동 스킬 슬롯
- 궁극기 슬롯
- Damage `GameplayEffect`
- Cooldown `GameplayEffect`
- 사망 관련 GameplayTag
- 서버 권한 피해 적용
- 플레이어와 구조물 대상 피해 처리 연결
- 디버그 로그 또는 임시 UI로 전투 상태 확인

### 1주차 완료 기준

- `BaseCharacter`에 ASC가 연결된다.
- Health와 MaxHealth가 존재한다.
- 기본 공격 Ability가 실행된다.
- Damage GameplayEffect로 Health가 감소한다.
- 체력 0 이하에서 사망 처리가 호출된다.
- 적 플레이어 또는 테스트 구조물에 피해를 줄 수 있다.

### 주요 Trello 카드

- `[W0] GASCombatPlanning 문서 작성`
- `[W1] BaseCharacter에 AbilitySystemComponent 연결`
- `[W1] Health / MaxHealth AttributeSet 구현`
- `[W1] 기본 공격 GameplayAbility 구현`
- `[W1] Damage GameplayEffect 구현`
- `[W1] 체력 0 이하 사망 처리`
- `[W2] 공격 스킬 / 이동 스킬 / 궁극기 슬롯 구조`
- `[BG] 캐릭터별 콘셉트 스킬 1개 적용`

### 주의점

- GAS 구조를 지나치게 복잡하게 만들지 않는다.
- 1~2주 MVP에서는 공통 캐릭터 1종과 기본 Ability 슬롯 구조 검증에 집중한다.
- 캐릭터별 복잡한 패시브, 정교한 밸런스, 궁극기 게이지는 후순위로 둔다.
- 피해 확정은 서버 기준으로 처리한다.

## D. 전장 / 구조물 / 목표물

### 목적

2라인 테스트맵, 방어타워, 넥서스, 구조물 피해 및 파괴 흐름을 만든다.

이 영역은 FieryTale을 단순 TPS 팀 데스매치가 아니라 AOS식 목표 게임으로 만드는 핵심이다.

### 주요 담당

- 2라인 회색 박스 테스트맵
- 팀별 스폰 지점 배치
- 방어타워 위치
- 넥서스 위치
- `TowerBase` 공통 구조
- 구조물 TeamId
- 구조물 Health
- 구조물 피해 수신
- 구조물 파괴 상태
- 넥서스 보호/취약 상태
- 타워 파괴 이벤트와 넥서스 취약화 연결
- TPS 시점에서 구조물 실루엣과 목표 방향 가독성 확인

### 1주차 완료 기준

- 2라인 블록아웃 맵이 존재한다.
- Team A/B 스폰 위치가 분리되어 있다.
- 테스트 구조물 또는 방어타워가 배치되어 있다.
- 구조물이 피해를 받는다.
- 구조물 체력이 0이 되면 파괴 상태가 된다.
- 넥서스 또는 테스트 목표 파괴가 승패 흐름으로 연결된다.

### 주요 Trello 카드

- `[W0] 2라인 테스트맵 블록아웃 규칙 정리`
- `[W1] 팀별 PlayerStart 스폰 구현`
- `[W1] TowerBase 공통 구조 구현`
- `[W1] 구조물 피해 및 파괴 처리`
- `[W2] 방어타워 파괴 시 넥서스 취약화`
- `[W2] 넥서스 피해 및 파괴`
- `[BG] 단순 미니언 웨이브 구현 검토`

### 주의점

- 2주 MVP에서는 미니언, 정글, 중립 오브젝트, 복잡한 우회로를 넣지 않는다.
- 타워 자동 공격은 필수가 아니다.
- 구조물 밸런스보다 상태 전이와 승패 연결을 우선한다.
- TPS 시점에서 타워와 넥서스가 멀리서도 구분되어야 한다.

## E. UI / 로비 / 데이터 / QA

### 목적

플레이어가 접속, 팀 상태, 캐릭터 선택, HUD, 승패 결과를 이해할 수 있게 한다. 동시에 QA 체크리스트와 시연 흐름을 정리한다.

이 영역은 2주 MVP를 시연 가능한 형태로 보이게 만드는 역할을 한다.

### 주요 담당

- 기본 로비 화면
- 접속 플레이어 목록
- 팀 배정 상태 표시
- Data Table 기반 캐릭터 목록
- 캐릭터 선택 요청
- 선택 캐릭터 표시
- HUD 생성
- 체력 표시
- 스킬 쿨다운 표시
- 팀 표시
- 사망/리스폰 상태 표시
- 타워/넥서스 상태 표시
- 승패 메시지
- QA 체크리스트
- 3~4주차 채팅, 킬로그, TAB 리더보드 확장

### 1주차 완료 기준

- 체력, 팀, 승패를 확인할 수 있는 최소 UI 또는 디버그 UI가 있다.
- 팀 배정 결과를 클라이언트에서 확인할 수 있다.
- 1주차 테스트 체크리스트가 존재한다.
- 한 판 테스트 시 확인해야 할 항목이 정리되어 있다.

### 2주차 완료 기준

- 기본 로비 화면이 있다.
- 접속 플레이어와 팀 상태를 볼 수 있다.
- Data Table 기반 캐릭터 선택 흐름이 작동한다.
- 선택한 캐릭터 데이터로 스폰된다.
- 승패 메시지가 표시된다.

### 주요 Trello 카드

- `[W2] 기본 로비 화면 구현`
- `[W2] 접속 플레이어 목록 표시`
- `[W2] Data Table 기반 캐릭터 선택`
- `[W2] HUD 체력/팀/구조물 상태 표시`
- `[W2] 승패 메시지 표시`
- `[W3] 실시간 채팅`
- `[W3] 채팅창 통합 킬로그/시스템 알림`
- `[W3] TAB 리더보드`
- `[W4] 최종 시연 체크리스트 작성`
- `[W4] 반복 플레이 안정성 테스트`

### 주의점

- UI는 그래픽 완성도보다 정보 전달을 우선한다.
- 2주 MVP에서는 채팅과 리더보드를 필수로 보지 않는다.
- 미니맵은 이번 4주 프로토타입에서 제외한다.
- 로비와 캐릭터 선택은 완성형 UX보다 서버 상태와 데이터 흐름 검증을 우선한다.

## 초기 착수 우선순위

현재 문서와 Trello 상태 기준으로 바로 착수할 우선순위는 다음이다.

1. C가 `GASCombatPlanning.md` 기준을 정리하고 GAS 구현 범위를 고정한다.
2. D가 2라인 블록아웃 규칙을 정리하고 테스트맵 배치 기준을 확정한다.
3. A가 팀 배정, 스폰, 매치 상태 흐름 구현을 시작한다.
4. B가 `BaseCharacter` 이동, 카메라, 조준 연결을 시작한다.
5. E가 HUD, 로비 최소 흐름, QA 체크리스트를 준비한다.

## 1주차 통합 목표

1주차 종료 시점에는 다음 흐름이 가능해야 한다.

1. 서버 또는 Listen Server를 실행한다.
2. 클라이언트 2명 이상이 접속한다.
3. 플레이어가 Team A/B로 나뉜다.
4. 팀별 스폰 지점에 배치된다.
5. 캐릭터가 이동하고 카메라로 조준한다.
6. 기본 공격으로 적 또는 구조물에 피해를 준다.
7. 체력이 감소하고 사망 상태가 발생한다.
8. 구조물이 피해를 받고 파괴된다.
9. 넥서스 또는 테스트 목표 파괴 시 승패가 판정된다.
