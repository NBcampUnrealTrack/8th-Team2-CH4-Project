# Trello Task Guidelines

## 목적

이 문서는 FieryTale 4주 프로토타입을 주니어급 프로그래머 5명이 진행할 때, 업무를 Trello 카드처럼 작은 단위로 나누고 자율적으로 분배하기 위한 기준을 정리한다.

핵심 목표는 다음과 같다.

- 큰 기능을 작은 작업 카드로 나눈다.
- 각 카드는 명확한 완료 조건을 가진다.
- 팀원은 스스로 카드를 가져가되, MVP 우선순위를 따른다.
- 막힌 작업은 빠르게 드러내고 재분배한다.
- 매주 동작하는 빌드를 만든다.

## 운영 원칙

작업 방식은 역할 고정 배정보다 카드 기반 자율 할당을 우선한다.

단, 완전 자율로 두지 않고 다음 원칙을 적용한다.

- 작업 카드는 0.5일~1일 안에 끝낼 수 있는 크기로 만든다.
- 한 카드는 하나의 명확한 결과만 가진다.
- 각 카드는 태그를 가진다.
- 각 태그에는 검수 또는 통합 담당자를 둔다.
- 완료 기준은 "코드 작성"이 아니라 "동작 확인"이다.
- 막힌 카드는 즉시 `Blocked`로 표시한다.
- 2시간 이상 막히면 혼자 붙잡지 않고 공유한다.
- 매일 종료 전에 완료 카드와 막힌 카드를 확인한다.

## 보드 컬럼

Trello 보드는 다음 컬럼을 기준으로 운영한다.

| 컬럼 | 의미 |
| --- | --- |
| Backlog | 아직 시작하지 않은 전체 작업 |
| This Week | 이번 주에 처리할 작업 |
| Ready | 바로 가져갈 수 있는 작업 |
| In Progress | 현재 작업 중 |
| Review | 구현 후 검수 또는 통합 대기 |
| Done | 동작 확인까지 끝난 작업 |
| Blocked | 진행이 막힌 작업 |

`Done`으로 이동하려면 반드시 완료 조건을 만족해야 한다.

## 카드 기본 형식

모든 카드는 아래 형식을 따른다.

```text
제목:

태그:

목적:

작업 내용:

완료 조건:

확인 방법:

관련 파일/블루프린트:

주의 사항:
```

### 제목

제목은 짧고 결과 중심으로 쓴다.

좋은 예:

- `BaseCharacter에 ASC 연결`
- `TowerBase 체력 감소 구현`
- `로비 플레이어 목록 표시`
- `넥서스 취약 상태 전환 구현`

나쁜 예:

- `GAS 작업`
- `UI 만들기`
- `타워 구현`
- `전투 시스템`

## 태그 규칙

카드는 하나 이상의 태그를 가진다.

기본 태그:

| 태그 | 의미 |
| --- | --- |
| `MVP` | 2주차까지 반드시 필요한 작업 |
| `BestGoal` | 필수는 아니지만 성공하면 좋은 목표 |
| `Polish` | 3~4주차 완성도 개선 작업 |
| `Character` | 캐릭터, 이동, 카메라, 입력 |
| `GAS` | Attribute, Ability, GameplayEffect, GameplayTag |
| `Structure` | TowerBase, 방어타워, 넥서스 |
| `Network` | 서버 권한, 복제, GameState, PlayerState |
| `Lobby` | 로비, 접속, 팀 배정, 매치 시작 |
| `UI` | HUD, 승패 메시지, 쿨다운 표시 |
| `Data` | Data Table, 캐릭터 데이터, 설정값 |
| `Chat` | 실시간 채팅, 킬로그, 시스템 알림 |
| `Leaderboard` | TAB 리더보드 |
| `Bug` | 버그 수정 |
| `Blocked` | 현재 진행 불가 |

`MVP` 태그가 붙은 작업은 다른 작업보다 우선한다.

`BestGoal` 태그는 4주 필수 성공 기준에는 포함하지 않는다. Dedicated Server 검증처럼 성공하면 좋지만 일정 리스크가 큰 작업에 사용한다.

## 우선순위

카드 우선순위는 다음 순서로 판단한다.

1. 2주 MVP를 막고 있는 작업
2. 다른 카드의 선행 조건이 되는 작업
3. 서버 권한 매치 흐름과 직접 연결된 작업
4. 4주차 시연 완성도를 높이는 작업
5. 선택 확장 작업

선택 확장 작업은 MVP 카드가 밀려 있을 때 가져가지 않는다.

## 카드 크기 기준

카드는 작게 쪼갠다.

적절한 크기:

- 반나절 안에 초안 구현 가능
- 하루 안에 동작 확인 가능
- 완료 조건이 3~5개 이하
- 한 명이 독립적으로 진행 가능

너무 큰 카드 예:

- `GAS 전투 시스템 구현`
- `로비 전체 구현`
- `캐릭터 선택 시스템 구현`
- `구조물 시스템 구현`

작게 나눈 예:

- `AttributeSet에 Health/MaxHealth 추가`
- `기본 공격 Ability 입력 바인딩`
- `Damage GameplayEffect 적용`
- `로비 플레이어 목록 표시`
- `Data Table에서 캐릭터 이름 로딩`
- `선택한 캐릭터 ID를 PlayerState에 저장`

## 완료 기준

카드는 다음 기준을 만족해야 `Done`으로 이동할 수 있다.

- 빌드 또는 에디터 실행이 깨지지 않는다.
- 담당자가 직접 동작을 확인했다.
- 완료 조건을 모두 만족한다.
- 필요한 경우 서버/클라이언트 양쪽에서 확인했다.
- 다른 시스템에 영향을 주는 변경은 Review를 거쳤다.
- 미완성 부분이 있으면 카드에 남은 이슈를 적었다.

코드를 작성했지만 아직 실행 확인을 하지 못했다면 `Review` 또는 `In Progress`에 둔다.

## 자율 할당 규칙

팀원은 `Ready` 컬럼에서 스스로 카드를 가져간다.

가져갈 때 기준:

- `MVP` 태그를 우선한다.
- 자신이 이해할 수 있는 크기의 카드를 고른다.
- 이미 진행 중인 카드가 있으면 새 카드를 가져가지 않는다.
- 막힌 카드를 해결할 수 있으면 새 기능보다 막힌 카드 지원을 우선한다.

카드를 가져가면 다음을 한다.

1. 카드를 `In Progress`로 옮긴다.
2. 담당자 이름을 카드에 추가한다.
3. 오늘 끝낼 수 있는 범위인지 확인한다.
4. 필요한 질문을 카드 댓글에 남긴다.

## 검수와 통합

각 태그에는 검수 또는 통합 담당자를 둔다.

권장 담당:

| 태그 | 검수/통합 담당 |
| --- | --- |
| `Network` | A |
| `Lobby` | A 또는 E |
| `Character` | B |
| `GAS` | C |
| `Structure` | D |
| `UI` | E |
| `Data` | C 또는 E |
| `Chat` | E |
| `Leaderboard` | E |

검수 담당자는 모든 작업을 대신 구현하지 않는다. 해당 태그의 작업이 기존 구조와 충돌하지 않는지 확인하고, 막힌 작업을 풀어주는 역할을 한다.

## Blocked 처리

다음 상황이면 카드를 `Blocked`로 옮긴다.

- 선행 작업이 아직 끝나지 않았다.
- 구현 방법을 몰라 2시간 이상 진행이 멈췄다.
- 서버/클라이언트 동작 차이를 해결하지 못했다.
- 다른 사람이 만든 기능과 충돌한다.
- 빌드 오류가 발생했지만 원인을 찾지 못했다.

`Blocked` 카드에는 반드시 다음을 적는다.

```text
막힌 이유:
시도한 방법:
필요한 도움:
관련 로그/스크린샷:
```

막힌 카드는 매일 회의에서 먼저 확인한다.

## 주차별 카드 묶음

### 1주차 카드 묶음

1주차 목표는 `BaseCharacter`와 구조물 피해 흐름이다.

필수 태그:

- `MVP`
- `Character`
- `GAS`
- `Structure`
- `Network`

카드 예시:

```text
제목: BaseCharacter 이동/카메라 입력 연결
태그: MVP, Character
목적: 플레이어가 테스트맵에서 캐릭터를 조작할 수 있게 한다.
완료 조건:
- 이동 입력이 동작한다.
- 카메라 회전이 동작한다.
- 서버 접속 상태에서도 위치가 복제된다.
확인 방법:
- 서버 1개와 클라이언트 1개에서 이동 확인
```

```text
제목: BaseCharacter에 AbilitySystemComponent 연결
태그: MVP, Character, GAS
목적: 캐릭터가 GAS Ability와 Attribute를 사용할 수 있게 한다.
완료 조건:
- ASC가 생성된다.
- 서버와 클라이언트에서 ASC가 유효하다.
- 기본 Ability 부여가 가능하다.
확인 방법:
- PIE에서 로그로 ASC와 Ability 부여 확인
```

```text
제목: AttributeSet에 Health/MaxHealth 추가
태그: MVP, GAS
목적: 피해와 사망 처리를 위한 기본 체력 Attribute를 만든다.
완료 조건:
- Health와 MaxHealth가 존재한다.
- 초기값이 적용된다.
- Health 변경이 확인된다.
확인 방법:
- 디버그 로그 또는 HUD 임시 표시로 확인
```

```text
제목: TowerBase 체력/피해 수신 구현
태그: MVP, Structure, Network
목적: 구조물이 서버 권한으로 피해를 받을 수 있게 한다.
완료 조건:
- TowerBase가 팀과 체력을 가진다.
- 공격을 받으면 서버에서 체력이 감소한다.
- 체력이 0이면 파괴 상태가 된다.
확인 방법:
- 클라이언트 공격으로 서버 구조물 체력 감소 확인
```

### 2주차 카드 묶음

2주차 목표는 기본 로비, Data Table 캐릭터 선택, 넥서스 승패 완성이다.

필수 태그:

- `MVP`
- `Lobby`
- `Data`
- `UI`
- `Structure`
- `Network`

카드 예시:

```text
제목: 로비 플레이어 목록 표시
태그: MVP, Lobby, UI, Network
목적: 접속한 플레이어를 로비에서 확인할 수 있게 한다.
완료 조건:
- 접속한 플레이어 ID가 표시된다.
- 팀 정보가 표시된다.
- 플레이어가 나가면 목록이 갱신된다.
확인 방법:
- 클라이언트 2개 접속 후 목록 확인
```

```text
제목: 캐릭터 Data Table 구조 생성
태그: MVP, Data, Character, GAS
목적: 캐릭터 선택에 필요한 데이터를 테이블로 관리한다.
완료 조건:
- 캐릭터 ID, 표시 이름, 클래스, Attribute, Ability 목록 필드가 있다.
- 최소 2개 캐릭터 데이터가 등록된다.
- 로비에서 데이터 목록을 읽을 수 있다.
확인 방법:
- 로비 UI 또는 로그로 2개 데이터 로딩 확인
```

```text
제목: 선택한 캐릭터 데이터로 스폰
태그: MVP, Lobby, Data, Network
목적: 로비에서 선택한 캐릭터가 인게임 스폰에 반영되게 한다.
완료 조건:
- 선택 캐릭터 ID가 PlayerState에 저장된다.
- 매치 시작 시 선택 데이터로 Pawn이 스폰된다.
- 서버와 클라이언트에서 동일하게 보인다.
확인 방법:
- 서로 다른 캐릭터 2종 선택 후 스폰 확인
```

```text
제목: 타워 파괴 시 넥서스 취약 상태 전환
태그: MVP, Structure, Network
목적: 방어타워 중 하나가 파괴되면 넥서스를 공격 가능하게 만든다.
완료 조건:
- 타워 파괴 이벤트가 발생한다.
- 해당 팀 넥서스가 취약 상태가 된다.
- 취약 전에는 넥서스가 피해를 받지 않는다.
- 취약 후에는 넥서스가 피해를 받는다.
확인 방법:
- 타워 파괴 전/후 넥서스 피해 가능 여부 확인
```

### 3주차 카드 묶음

3주차 목표는 시연 완성도를 높이는 온라인 기능과 정보 표시다.

우선 태그:

- `Chat`
- `Leaderboard`
- `UI`
- `Polish`

카드 예시:

- `전체 채팅 송수신 구현`
- `채팅창에 킬로그 표시`
- `채팅창에 타워 파괴 알림 표시`
- `TAB 리더보드 열기/닫기`
- `리더보드에 킬/데스 표시`
- `로비 준비 상태 표시`

### 4주차 카드 묶음

4주차 목표는 새 기능보다 완성도 개선이다.

우선 태그:

- `Polish`
- `Bug`
- `UI`
- `Network`
- `BestGoal`

카드 예시:

- `공격 적중 피드백 추가`
- `피격 피드백 추가`
- `사망/리스폰 메시지 개선`
- `타워 파괴 피드백 추가`
- `승패 메시지 가독성 개선`
- `2 vs 2 접속 안정성 테스트`
- `반복 플레이 중 발생하는 크래시 수정`
- `Dedicated Server 빌드 생성`
- `Dedicated Server에 클라이언트 접속 확인`

`Dedicated Server` 관련 카드는 `BestGoal, Network` 태그를 붙인다. Listen Server 기준 2 vs 2 시연이 안정화되기 전에는 Dedicated Server 카드에 많은 시간을 쓰지 않는다.

## 일일 운영 방식

매일 시작할 때 확인할 것:

1. 오늘 반드시 끝내야 하는 `MVP` 카드
2. `Blocked` 카드
3. 어제 `Review`에 남은 카드
4. 각자 가져갈 카드

매일 종료할 때 확인할 것:

1. `Done`으로 옮긴 카드가 실제로 동작 확인됐는지
2. `In Progress`에 남은 카드 이유
3. `Blocked` 카드의 도움 요청
4. 다음 날 우선 처리할 카드

## 회의 규칙

회의는 짧게 한다.

각자 다음 3가지만 말한다.

1. 어제 완료한 카드
2. 오늘 가져갈 카드
3. 막힌 카드 또는 필요한 도움

기획 토론이나 큰 구조 변경은 회의 중 길게 하지 않고, 별도 카드나 짧은 기술 검토로 분리한다.

## 금지할 운영 방식

다음 방식은 피한다.

- 한 사람이 큰 기능 하나를 통째로 오래 들고 있기
- 완료 조건이 없는 카드 만들기
- 동작 확인 없이 `Done` 처리하기
- MVP 카드가 남아 있는데 선택 기능 먼저 작업하기
- 막힌 작업을 공유하지 않고 하루 이상 붙잡기
- 4주차에 큰 신규 시스템 추가하기

## 최종 기준

이 방식의 성공 기준은 카드 수를 많이 처리하는 것이 아니다.

성공 기준은 다음이다.

- 매주 동작하는 빌드가 있다.
- 2주차에 MVP 한 판이 끝까지 돌아간다.
- 4주차에 2 vs 2 시연이 가능하다.
- 각 카드의 완료 조건이 실제 플레이에서 확인된다.
- 팀원이 자신이 가져갈 작업을 스스로 판단할 수 있다.
