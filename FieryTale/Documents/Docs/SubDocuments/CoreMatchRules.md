# Core Match Rules

## 목적

이 문서는 FieryTale 4주 프로토타입에서 한 판의 시작, 진행, 종료 규칙을 구현 가능한 형태로 고정하기 위한 기준 문서다.

목표는 기획 의도를 넓게 설명하는 것이 아니라, Unreal Engine 구현자가 `GameMode`, `GameState`, `PlayerState`, `PlayerController`, 구조물 Actor, HUD로 바로 분해할 수 있는 최소 규칙을 정리하는 것이다.

## 현재 결정

| 항목 | 결정 |
|---|---|
| 기본 장르 축 | TPS 팀 전투와 단순 AOS 목표의 절충형 |
| 매치 규모 | 2 vs 2 |
| 최소 검증 규모 | 1 vs 1로도 한 판 완료 가능 |
| 팀 수 | 2개 |
| 시점 | TPS |
| 전장 | 2라인 압축형 전장 |
| 승리 조건 | 상대 팀 넥서스 파괴 |
| 미니언 | 2주 MVP 제외 |
| 성장 | 1차 MVP 제외 |
| 매치 제한 시간 | 2주 MVP 제외 |
| 서버 기준 | Listen Server 필수, Dedicated Server 선택 목표 |
| 권한 구조 | 피해, 사망, 리스폰, 구조물 파괴, 승패는 서버 권한 |

## 매치 구성

### 팀

MVP 매치는 두 팀으로 구성한다.

| 팀 | 임시 이름 | 설명 |
|---|---|---|
| Team A | Blue | 블루 팀, 임시 기준 |
| Team B | Red | 레드 팀, 임시 기준 |

팀 이름은 임시값이다. 구현에서는 숫자형 `TeamId`를 기준으로 처리하고, UI 표시명은 나중에 교체할 수 있게 둔다.

기본 규칙:

- 한 매치에는 항상 2개의 팀이 있다.
- 각 플레이어는 서버에서 하나의 팀을 배정받는다.
- 2 vs 2가 최종 시연 목표지만, 1 vs 1에서도 매치가 시작되고 종료될 수 있어야 한다.
- 팀 배정 결과는 `PlayerState` 또는 동등한 복제 상태에 저장한다.
- 클라이언트는 자기 팀을 직접 확정하지 않는다.

### 플레이어 수

필수 목표:

- 1 vs 1 매치 시작 가능
- 2 vs 2 매치 시작 가능

MVP에서는 다음 기능을 필수로 보지 않는다.

- 관전자
- 중도 참가
- 중도 이탈 보정
- 자동 매치메이킹
- 팀 교체 UI
- 랭크, 파티, 큐 시스템

중도 이탈이 발생하면 2주 MVP에서는 별도 패널티 없이 남은 플레이어가 계속 진행할 수 있게 둔다. 단, 이탈 처리로 서버가 멈추거나 승패 판정이 깨지면 안 된다.

## 매치 상태 흐름

매치는 다음 상태를 기준으로 관리한다.

| 상태 | 설명 | 주요 처리 |
|---|---|---|
| Lobby | 플레이어 접속과 준비 상태 확인 | 팀 배정, 캐릭터 선택, 시작 대기 |
| Starting | 매치 시작 준비 | 스폰 위치 결정, 상태 초기화 |
| InProgress | 실제 플레이 진행 | 전투, 사망, 리스폰, 구조물 피해 |
| Ending | 승패 확정 직후 | 입력 제한, 결과 표시 |
| Finished | 매치 종료 | 재시작 또는 로비 복귀 대기 |

`GameMode`는 서버 전용 매치 상태 전이를 관리한다. `GameState`는 클라이언트가 UI로 표시해야 하는 현재 매치 상태, 남은 준비 상태, 승패 결과를 복제한다.

### 상태 전이

기본 흐름:

1. 서버 또는 호스트가 매치를 연다.
2. 플레이어가 로비에 접속한다.
3. 서버가 플레이어에게 팀을 배정한다.
4. 플레이어가 캐릭터를 선택한다.
5. 시작 조건을 만족하면 매치가 `Starting`으로 전환된다.
6. 모든 플레이어와 구조물 상태를 초기화한다.
7. 팀별 스폰 위치에 플레이어를 배치한다.
8. 매치가 `InProgress`로 전환된다.
9. 한 팀의 넥서스가 파괴되면 `Ending`으로 전환된다.
10. 승패 메시지를 표시한 뒤 `Finished`로 전환된다.

2주 MVP에서는 시작 조건을 단순하게 둔다.

- 호스트 또는 서버 권한자가 시작 명령을 내리면 시작한다.
- 최소 2명 이상 접속 시 1 vs 1로 시작 가능하다.
- 4명 접속 시 2 vs 2로 시작 가능하다.
- 준비 완료 UI가 미완성이라면 디버그 시작 명령으로 대체할 수 있다.

## 로비 규칙

로비의 목적은 상용 매치메이킹이 아니라, 4주 프로토타입에서 매치 시작 전 최소 상태를 보여주는 것이다.

필수 표시:

- 접속한 플레이어 목록
- 각 플레이어의 팀
- 각 플레이어의 선택 캐릭터
- 매치 시작 가능 여부

필수 처리:

- 플레이어 접속 시 서버가 팀을 배정한다.
- 플레이어는 Data Table에 등록된 캐릭터 중 하나를 선택한다.
- 선택 결과는 서버가 검증하고 저장한다.
- 매치 시작 시 선택된 캐릭터 데이터로 Pawn 또는 Character를 스폰한다.

2주 MVP에서 캐릭터 선택 UI가 늦어질 경우, 서버 기본값으로 캐릭터를 지정해도 된다. 다만 Data Table 기반 선택 구조 자체는 2주차 목표에 포함한다.

## 팀 배정 규칙

팀 배정은 서버가 처리한다.

초기 규칙:

1. 새 플레이어가 접속한다.
2. 서버가 현재 인원이 적은 팀을 찾는다.
3. 인원이 적은 팀에 플레이어를 배정한다.
4. 양 팀 인원이 같으면 Team A, Team B 순서로 번갈아 배정한다.

예시:

| 접속 순서 | 배정 팀 |
|---|---|
| 1번째 플레이어 | Team A |
| 2번째 플레이어 | Team B |
| 3번째 플레이어 | Team A |
| 4번째 플레이어 | Team B |

MVP에서는 플레이어가 직접 팀을 고르는 기능을 필수로 넣지 않는다. 팀 선택은 테스트 편의가 필요할 때만 디버그 기능으로 둔다.

## 스폰 규칙

### 시작 스폰

매치 시작 시 각 플레이어는 자기 팀의 스폰 지점에 배치된다.

필수 기준:

- 각 팀은 최소 1개의 스폰 지점을 가진다.
- 2 vs 2에서는 같은 팀 플레이어가 겹치지 않도록 스폰 위치를 약간 분리한다.
- 스폰 위치는 넥서스 근처 또는 팀 시작 구역에 둔다.
- 상대 팀이 시작 직후 바로 공격할 수 없는 거리를 확보한다.

시작 스폰 처리:

1. 서버가 플레이어의 팀을 확인한다.
2. 서버가 해당 팀의 사용 가능한 스폰 지점을 고른다.
3. 서버가 선택된 캐릭터 데이터로 Pawn 또는 Character를 생성한다.
4. Controller를 Pawn에 Possess한다.
5. 체력과 Ability 상태를 초기화한다.
6. 클라이언트 HUD가 팀, 체력, 스킬 상태를 표시한다.

### 리스폰

플레이어가 사망하면 고정 시간 뒤 자기 팀 스폰 지점에서 부활한다.

초기 리스폰 시간:

| 항목 | 값 |
|---|---|
| 리스폰 대기 | 5초 |
| 체력 회복 | MaxHealth까지 회복 |
| 스킬 쿨다운 | 기존 쿨다운 유지 또는 초기화 중 구현 단순한 쪽 선택 |
| 위치 | 자기 팀 스폰 지점 |

2주 MVP에서는 사망 횟수에 따라 리스폰 시간이 증가하지 않는다.

리스폰 처리:

1. 서버가 체력 0 이하를 확인한다.
2. 플레이어를 사망 상태로 전환한다.
3. 입력, 이동, 공격, 스킬 사용을 막는다.
4. 사망 상태와 리스폰 남은 시간을 복제한다.
5. 5초 뒤 자기 팀 스폰 지점으로 이동시킨다.
6. 체력을 회복한다.
7. 사망 상태를 해제하고 조작 가능 상태로 전환한다.

## 전투 기본 규칙

전투 판정은 서버 권한을 기준으로 한다.

필수 기준:

- 클라이언트 입력은 요청이다.
- 실제 피해 적용은 서버가 결정한다.
- Health, Death, Respawn 상태는 서버 상태를 기준으로 복제한다.
- 기본 공격과 스킬은 자기 팀을 피해 대상으로 삼지 않는다.
- 적 플레이어와 적 구조물은 공격 대상이 될 수 있다.

2주 MVP의 공격 대상:

| 대상 | 공격 가능 여부 |
|---|---|
| 적 플레이어 | 가능 |
| 아군 플레이어 | 불가 |
| 적 방어타워 | 가능 |
| 아군 방어타워 | 불가 |
| 취약 상태의 적 넥서스 | 가능 |
| 보호 상태의 적 넥서스 | 피해 불가 |
| 아군 넥서스 | 불가 |
| 미니언 | MVP 제외 |

TPS 기준 기본 공격은 카메라 전방 또는 조준점 기준으로 발동한다. 단, 조준 보정과 정교한 클라이언트 예측은 1~2주 MVP 필수 범위에 넣지 않는다.

## 구조물 규칙

MVP 구조물은 방어타워와 넥서스로 나눈다.

| 구조물 | 팀별 수 | 목적 |
|---|---:|---|
| 방어타워 | 2개 | 각 라인의 1차 목표 |
| 넥서스 | 1개 | 최종 승패 목표 |

구조물은 다음 공통 상태를 가진다.

- TeamId
- CurrentHealth
- MaxHealth
- IsDestroyed
- IsDamageable
- TargetCollision
- 서버 권한 피해 처리
- 파괴 이벤트
- HUD 표시용 상태

### 방어타워

방어타워는 각 팀의 라인 방어 구조물이다.

필수 규칙:

- 각 팀은 방어타워 2개를 가진다.
- 각 방어타워는 서로 다른 라인에 배치한다.
- 적 플레이어의 공격으로 피해를 받는다.
- 체력이 0이 되면 파괴된다.
- 파괴 상태는 복제되어 모든 클라이언트가 확인할 수 있어야 한다.
- 방어타워 중 하나라도 파괴되면 해당 팀의 넥서스가 취약 상태가 된다.

2주 MVP에서는 방어타워 자동 공격을 필수로 구현하지 않는다.

후순위 확장:

- 적 플레이어 자동 공격
- 공격 범위
- 라인 미니언 우선 공격
- 구조물 투사체
- 타워 경고 UI

### 넥서스

넥서스는 각 팀의 최종 목표 구조물이다.

필수 규칙:

- 각 팀은 넥서스 1개를 가진다.
- 넥서스는 기본 상태에서 보호 상태다.
- 보호 상태의 넥서스는 피해를 받지 않는다.
- 같은 팀 방어타워 중 하나라도 파괴되면 넥서스는 취약 상태가 된다.
- 취약 상태의 넥서스는 적 플레이어의 공격으로 피해를 받는다.
- 넥서스 체력이 0이 되면 해당 팀은 패배한다.

넥서스 취약화 조건:

```text
Team A 방어타워 2개 중 1개 이상 파괴됨
=> Team A 넥서스 취약화
=> Team B가 Team A 넥서스에 피해를 줄 수 있음
```

이 규칙은 2 vs 2 소규모 전장에서 한 라인만 성공적으로 밀어도 게임이 전진하도록 하기 위한 것이다.

## 라인 규칙

MVP 전장은 2개의 주요 라인을 가진다.

라인의 목적:

- 플레이어가 공격 경로를 이해하게 한다.
- 방어타워 위치를 기준으로 전투 목표를 만든다.
- 넥서스로 향하는 경로를 명확하게 만든다.

MVP 라인은 복잡한 운영보다 이동과 교전을 읽기 쉽게 만드는 것이 중요하다.

필수 기준:

- 각 라인에는 양 팀의 방어타워가 하나씩 배치된다.
- 라인은 상대 넥서스 방향을 직관적으로 읽을 수 있어야 한다.
- TPS 시점에서 타워와 넥서스 실루엣이 멀리서도 구분되어야 한다.
- 라인 사이 우회 공간은 있어도 되지만, 2주 MVP에서는 단순해야 한다.

2주 MVP에서는 다음 요소를 제외한다.

- 미니언 웨이브
- 정글
- 중립 몬스터
- 중립 오브젝트
- 시야 와드
- 복잡한 수직 지형

## 사망 규칙

플레이어 체력이 0 이하가 되면 사망한다.

사망 시 처리:

- 캐릭터 입력 비활성화
- 공격과 스킬 사용 비활성화
- 충돌 처리 단순화
- 사망 상태 표시
- 리스폰 타이머 시작
- 킬/데스 기록 갱신

킬/데스는 `PlayerState`에 저장한다.

2주 MVP에서는 어시스트, 피해량, 구조물 피해량은 필수로 보지 않는다. 3~4주차 리더보드 확장 시 추가한다.

## 승패 규칙

승패는 넥서스 파괴로만 결정한다.

승리 조건:

- 상대 팀 넥서스의 체력을 0 이하로 만든 팀이 승리한다.

패배 조건:

- 자기 팀 넥서스 체력이 0 이하가 되면 패배한다.

2주 MVP에서는 다음 승패 조건을 사용하지 않는다.

- 제한 시간 종료
- 킬 수 비교
- 남은 타워 수 비교
- 넥서스 체력 비교
- 점수제
- 항복

승패 처리:

1. 서버가 넥서스 체력 0 이하를 확인한다.
2. 서버가 패배 팀과 승리 팀을 결정한다.
3. 매치 상태를 `Ending`으로 전환한다.
4. 모든 플레이어의 공격과 스킬 사용을 막는다.
5. `GameState`에 승패 결과를 복제한다.
6. 클라이언트 HUD에 승패 메시지를 표시한다.
7. 일정 시간 뒤 `Finished` 상태로 전환한다.

초기 종료 대기 시간은 5초를 기준 후보로 둔다.

## HUD 표시 규칙

2주 MVP 필수 HUD:

- 자기 체력
- 스킬 쿨다운 4개
- 자기 팀
- 사망/리스폰 상태
- 아군/적 방어타워 상태
- 아군/적 넥서스 상태
- 넥서스 취약화 여부
- 승패 메시지

3~4주차 확장 HUD:

- TAB 리더보드
- 킬로그
- 채팅창 통합 시스템 알림
- 접속 상태
- 구조물 파괴 알림

미니맵은 이번 4주 프로토타입에서 제외한다.

## 서버 권한 기준

서버가 확정해야 하는 항목:

- 팀 배정
- 캐릭터 선택 결과 검증
- 매치 시작과 종료
- 스폰과 리스폰
- 피해 적용
- 사망 판정
- 구조물 파괴
- 넥서스 취약화
- 승패 판정
- 킬/데스 기록

클라이언트가 담당하는 항목:

- 입력 전송
- 카메라 조작
- 로컬 HUD 표시
- 서버에서 복제된 상태 표시
- 이펙트와 사운드 재생 요청 또는 표시

클라이언트는 체력, 사망, 구조물 파괴, 승패를 직접 확정하지 않는다.

## Unreal 구현 분해 기준

### GameMode

서버 전용 규칙을 담당한다.

- 플레이어 접속 처리
- 팀 배정
- 매치 상태 전이
- 시작 조건 확인
- 스폰과 리스폰 호출
- 넥서스 파괴 시 승패 결정

### GameState

클라이언트가 알아야 하는 매치 상태를 복제한다.

- 현재 매치 상태
- 팀별 구조물 상태 요약
- 승리 팀
- 매치 진행 여부
- 로비 플레이어 목록 표시용 데이터

### PlayerState

플레이어별 지속 상태를 가진다.

- PlayerId 또는 임시 이름
- TeamId
- 선택 캐릭터 ID
- Kill
- Death
- 현재 접속 상태

### PlayerController

입력과 UI 연결을 담당한다.

- 로비 입력
- 캐릭터 선택 요청
- 매치 시작 요청
- HUD 생성
- 리스폰 상태 표시

### Character 또는 Pawn

실제 조작 캐릭터를 담당한다.

- 이동
- TPS 카메라
- 조준 방향
- AbilitySystemComponent
- AttributeSet
- 사망 상태
- 서버 권한 공격 요청

### Structure Actor

방어타워와 넥서스의 공통 구조를 담당한다.

- TeamId
- Health
- Damageable 상태
- Destroyed 상태
- 피해 수신
- 파괴 이벤트
- HUD 표시용 상태 제공

## MVP 테스트 체크리스트

2주 MVP 완료 전 다음 항목을 확인한다.

| 항목 | 완료 기준 |
|---|---|
| 1 vs 1 시작 | 두 클라이언트가 서로 다른 팀으로 스폰된다. |
| 2 vs 2 시작 | 네 클라이언트가 2명씩 팀에 배정된다. |
| 팀 표시 | 각 클라이언트가 자기 팀을 확인할 수 있다. |
| 기본 공격 | 적 플레이어와 적 구조물에 피해를 준다. |
| 아군 피해 방지 | 아군 플레이어와 아군 구조물에는 피해가 들어가지 않는다. |
| 사망 | 체력 0 이하에서 입력과 공격이 막힌다. |
| 리스폰 | 5초 뒤 자기 팀 스폰 지점에서 부활한다. |
| 타워 파괴 | 타워 체력 0 이하에서 파괴 상태가 복제된다. |
| 넥서스 보호 | 타워가 남아 있으면 넥서스 피해가 막힌다. |
| 넥서스 취약화 | 타워 1개 이상 파괴 후 넥서스 피해가 가능하다. |
| 승패 | 넥서스 파괴 시 승리 팀과 패배 팀이 표시된다. |
| 매치 종료 | 승패 이후 추가 피해와 스킬 사용이 막힌다. |
