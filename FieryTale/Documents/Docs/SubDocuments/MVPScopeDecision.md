# MVP Scope Decision

## 목적

이 문서는 FieryTale의 4주 프로토타입에서 그래픽 요소를 제외하고, 어떤 규모의 게임 플레이를 MVP로 만들지 결정하기 위한 기준 문서다.

현재 목표는 4주차에 어느 정도 완성된 프로토타입처럼 보이는 상태를 만드는 것이다. 따라서 MVP 자체는 2주 이내에 완성하고, 남은 2주는 조작감, 피드백, 안정성, 밸런스, UI, 사운드 등 폴리싱에 사용한다.

## 4주 목표

4주 프로젝트의 기준은 다음과 같다.

- 1~2주차: 게임 한 판이 끝까지 돌아가는 MVP 완성
- 3~4주차: 폴리싱, 밸런스, UX 피드백, 버그 수정, 선택 기능 추가
- 그래픽 완성도보다 규칙의 명확함, 플레이 흐름, 서버 안정성, 전투 피드백을 우선한다.

## MVP 장르 축

MVP는 순수 AOS도, 순수 TPS 팀 데스매치도 아닌 절충형으로 잡는다.

기준 방향은 다음과 같다.

- TPS 팀 전투를 기본 조작 감각으로 사용한다.
- AOS식 승리 목표를 단순화해서 사용한다.
- 플레이어는 상대 팀과 교전하면서 방어 구조물과 넥서스를 파괴한다.
- 복잡한 성장, 아이템, 정글, 넓은 우회 동선은 MVP 필수 범위에서 제외한다.

## 네트워크 기준

FieryTale은 장기적으로 Dedicated Server 기반 멀티플레이를 목표로 한다.

다만 4주 프로토타입에서는 주니어급 5인 팀과 제한된 일정을 고려해, 개발과 필수 성공 기준은 Listen Server 중심으로 잡는다.

기준은 다음과 같다.

- 서버 권한 구조는 처음부터 지킨다.
- 개발 중 기본 테스트는 Listen Server로 진행한다.
- 2주 MVP 완료 기준은 Listen Server에서 한 판이 끝까지 돌아가는 것이다.
- 4주차 최종 목표도 Listen Server 기준 2 vs 2 시연 가능 상태를 필수로 본다.
- Dedicated Server 실행 검증은 베스트 목표로 둔다.

### Listen Server 필수 범위

4주 프로토타입에서 반드시 확인해야 할 항목은 다음과 같다.

- 호스트가 방 또는 서버를 연다.
- 다른 클라이언트가 접속한다.
- 팀 배정이 서버 권한으로 처리된다.
- 캐릭터 선택 결과가 서버 기준으로 적용된다.
- 피해, 사망, 리스폰, 구조물 파괴가 서버 권한으로 처리된다.
- 넥서스 파괴와 승패 판정이 서버 권한으로 처리된다.
- 2 vs 2 시연이 가능하다.

### Dedicated Server 선택 목표

Dedicated Server는 가능하면 4주차에 검증한다.

선택 목표:

- Dedicated Server 빌드 생성
- 서버 프로세스 실행
- 클라이언트 접속
- 로비 또는 매치 진입
- 최소 1 vs 1 플레이 확인
- 가능하면 2 vs 2 플레이 확인

Dedicated Server 검증이 실패하더라도, Listen Server에서 서버 권한 구조가 정상 작동하고 2 vs 2 시연이 가능하면 4주 프로토타입의 필수 목표는 달성한 것으로 본다.

### 구현 주의점

Dedicated Server를 나중에 붙일 수 있도록 다음 기준은 처음부터 지킨다.

- 중요한 게임 판정은 클라이언트에서 확정하지 않는다.
- 체력, 피해, 사망, 구조물 파괴, 승패는 서버가 결정한다.
- GameMode에는 서버 전용 규칙을 둔다.
- GameState와 PlayerState에는 클라이언트가 알아야 할 복제 상태를 둔다.
- UI는 서버 상태를 직접 결정하지 않고, 복제된 상태를 표시한다.
- 로컬 테스트 편의 기능이 서버 권한 규칙을 깨지 않게 한다.

## 전장 구조

MVP 전장은 작은 규모의 2라인 구조로 잡는다.

필수 구조:

- 팀 2개
- 라인 2개
- 팀별 넥서스 1개
- 팀별 1단계 방어타워
- 팀별 스폰 지점

낮은 우선순위:

- 큰 맵
- 복잡한 우회로
- 정글 지역
- 중립 오브젝트
- 다단계 타워 구조
- 고도 차이가 큰 복잡한 전장

2주 MVP에서는 전장 크기를 키우기보다, 플레이어가 빠르게 만나고 구조물 목표를 이해할 수 있는 압축된 맵을 우선한다.

## 미니언 범위

미니언은 2주 MVP 필수 범위에 넣지 않는다.

우선순위는 다음과 같다.

1. 1~2주차에는 미니언 없이 플레이어가 직접 라인을 밀고 구조물을 공격하는 형태로 한 판의 시작과 종료를 완성한다.
2. 3주차 이후 여유가 있으면 단순 미니언 웨이브를 추가한다.
3. 미니언이 추가될 경우 Deadlock 스타일을 참고해 근거리, 원거리, 힐러 미니언 3종을 기본 후보로 둔다.

미니언을 뒤로 미루는 이유는 다음과 같다.

- 2주 MVP의 핵심은 서버 권한 매치 흐름, 전투, 구조물 파괴, 승패 판정이다.
- 미니언은 AI, 경로 이동, 타겟팅, 공격 판정, 네트워크 동기화, 밸런스 비용을 추가한다.
- 미니언이 없어도 라인, 타워, 넥서스를 통해 최소 AOS 목표 구조는 검증할 수 있다.

## 미니언 추가 시 기본 설계

3주차 이후 미니언을 추가한다면, 첫 구현은 단순해야 한다.

### 근거리 미니언

역할:

- 라인의 기본 전방 유닛
- 구조물과 적 미니언을 먼저 받아내는 역할

기본 행동:

- 정해진 라인을 따라 이동한다.
- 가장 가까운 적 미니언, 플레이어, 구조물을 공격한다.
- 짧은 사거리의 근접 공격을 사용한다.

### 원거리 미니언

역할:

- 후방에서 지속 피해를 주는 유닛
- 라인 압박 속도를 높이는 역할

기본 행동:

- 근거리 미니언보다 뒤에서 이동한다.
- 일정 사거리 안의 적을 투사체 또는 히트스캔으로 공격한다.
- 체력은 근거리 미니언보다 낮게 둔다.

### 힐러 미니언

역할:

- 아군 웨이브 유지력을 올리는 지원 유닛
- 라인 싸움에 변수를 주는 유닛

기본 행동:

- 아군 미니언 또는 구조물 근처를 따라간다.
- 일정 주기로 체력이 낮은 아군 미니언을 회복한다.
- 초기 구현에서는 플레이어 회복은 제외할 수 있다.

## 2주 MVP 필수 완료 조건

2주차 종료 시점에는 다음이 가능해야 한다.

1. 서버 실행
2. 2명 이상 클라이언트 접속
3. 팀 배정과 팀별 스폰
4. TPS 이동, 조준, 기본 공격
5. 체력 감소, 사망, 리스폰
6. 방어타워 피해 및 파괴
7. 넥서스 피해 및 파괴
8. 넥서스 파괴 시 승패 판정
9. 한 판 재시작 또는 종료 흐름

## 3~4주차 폴리싱 목표

3~4주차에는 새 시스템을 늘리기보다 이미 있는 한 판의 완성도를 높인다.

우선 폴리싱 대상:

- 공격 피드백
- 피격 피드백
- 사망과 리스폰 흐름
- 구조물 피격/파괴 피드백
- 승패 UI
- 기본 HUD
- 서버 접속 안정성
- 조작감
- 기본 밸런스
- 버그 수정

선택 확장 대상:

- 단순 미니언 웨이브
- 근거리/원거리/힐러 미니언 3종
- 캐릭터별 스킬 1~2개 추가
- 간단한 매치 타이머
- 라인별 알림

## 캐릭터 제작 범위

1주차에는 여러 캐릭터를 동시에 만들지 않고, 공통 기반이 되는 `BaseCharacter` 1종을 먼저 완성한다.

`BaseCharacter`의 목적은 최종 캐릭터 하나를 만드는 것이 아니라, 이후 동화 캐릭터들을 빠르게 파생 제작할 수 있는 상속 구조와 GAS 연결 구조를 검증하는 것이다.

### 1주차 BaseCharacter 목표

1주차에 완성해야 할 공통 기능은 다음과 같다.

- 이동
- 카메라
- 조준
- 팀 구분
- 체력 Attribute
- 기본 공격 Ability
- 공격 스킬 Ability 슬롯
- 이동 스킬 Ability 슬롯
- 궁극기 Ability 슬롯
- 피해 GameplayEffect
- 사망 처리
- 리스폰 처리
- 구조물 공격 가능 여부
- 서버 권한 판정

이 단계에서는 캐릭터 고유 패시브, 복잡한 애니메이션, 캐릭터별 전용 UI, 세밀한 밸런스는 필수 범위에서 제외한다.

### GAS 적용 방향

GAS를 사용하는 이유는 캐릭터가 늘어날수록 강점이 커지는 구조를 만들기 위해서다.

`BaseCharacter`는 다음 공통 요소를 가진다.

- AbilitySystemComponent
- 공통 AttributeSet
- 기본 GameplayTag 규칙
- 기본 Ability 입력 바인딩
- 피해 처리 흐름
- 사망/리스폰 상태 처리

파생 캐릭터는 다음 요소를 교체하거나 추가하는 방식으로 만든다.

- 시작 Attribute 값
- 부여받는 GameplayAbility 목록
- GameplayEffect 수치
- GameplayTag 조합
- 공격 방식
- 스킬 쿨다운
- 역할군 데이터

이 구조를 사용하면 캐릭터별 코드를 매번 새로 작성하기보다, 공통 전투 프레임워크 위에서 데이터와 Ability 조합을 바꾸는 방식으로 캐릭터를 확장할 수 있다.

### 4주 프로토타입 기준

4주 프로토타입에서 캐릭터 목표는 다음처럼 잡는다.

- 1주차: `BaseCharacter` 1종과 기본 Ability 슬롯 구조 완성
- 2주차: `BaseCharacter`로 한 판의 MVP 완료
- 3주차 이후: 파생 캐릭터 1~2종 추가 검토
- 4주차: 캐릭터 수를 늘리기보다 이미 있는 캐릭터의 조작감, 피드백, 밸런스 개선 우선

파생 캐릭터가 추가될 경우에도, 목표는 많은 캐릭터 수가 아니라 "상속 구조와 GAS 기반 확장이 실제로 빠르게 되는지"를 증명하는 것이다.

## 스킬 범위

최종적으로 `BaseCharacter`와 파생 캐릭터는 다음 4개 전투 입력을 기준으로 설계한다.

- 기본 공격
- 공격 스킬
- 이동 스킬
- 궁극기

2주 MVP에서 중요한 것은 수치 밸런스가 아니라, 각 스킬 타입이 GAS를 통해 정상적으로 실행되고 서버 권한으로 판정되는지 확인하는 것이다.

### 기본 공격

목적:

- 가장 기본적인 피해 처리 흐름 검증
- 플레이어와 구조물을 모두 공격할 수 있는지 확인

검증 항목:

- Ability 입력
- 서버 권한 피해 적용
- GameplayEffect 적용
- 체력 감소
- 사망 또는 구조물 파괴 연결

### 공격 스킬

목적:

- 기본 공격보다 강하거나 다른 판정 방식을 가진 공격 Ability 검증
- 파생 캐릭터가 다른 공격 방식을 가질 수 있는지 확인

초기 구현 예시:

- 직선 투사체
- 짧은 범위 공격
- 히트스캔 강화 공격

2주 MVP에서는 하나의 단순한 형태만 구현해도 된다.

### 이동 스킬

목적:

- TPS 전투에서 위치 변경과 생존 수단 검증
- Ability 실행 중 이동, 쿨다운, 서버 동기화 흐름 확인

초기 구현 예시:

- 짧은 대시
- 뒤로 회피
- 일정 거리 순간 가속

복잡한 지형 판정, 벽 넘기, 공중 이동은 MVP 필수 범위에서 제외한다.

### 궁극기

목적:

- 긴 쿨다운 또는 강한 효과를 가진 Ability 슬롯 검증
- 캐릭터별 차별화 여지를 보여주는 상위 스킬 구조 확인

초기 구현 예시:

- 강한 범위 피해
- 긴 사거리 투사체
- 짧은 시간 자기 강화

2주 MVP에서는 궁극기 게이지나 정교한 조건부 발동보다, 입력 시 긴 쿨다운을 가진 강한 Ability가 실행되는 구조를 우선한다.

### 밸런스 기준

현재 단계에서는 밸런스를 핵심 목표로 두지 않는다.

따라서 다음 항목은 4주차 폴리싱 또는 이후 단계에서 다룬다.

- 캐릭터 간 승률
- 스킬별 DPS
- 역할군 간 상성
- 정교한 쿨다운 수치
- 궁극기 충전 속도
- 미니언과 구조물 대상 피해 보정

이번 MVP에서 필요한 밸런스 기준은 "플레이가 즉시 깨지지 않을 정도"로 제한한다.

## 구조물과 승패 규칙

MVP의 승리 조건은 상대 팀 넥서스 파괴다.

단, 넥서스는 처음부터 공격 가능한 상태가 아니다. 각 팀의 1단계 방어타워 중 하나라도 파괴되면 해당 팀의 넥서스가 공격 가능한 상태가 된다.

기본 흐름은 다음과 같다.

1. 플레이어는 두 라인 중 하나를 통해 상대 진영으로 진입한다.
2. 각 라인에는 상대 팀의 1단계 방어타워가 있다.
3. 방어타워 중 하나라도 파괴되면 상대 넥서스가 취약 상태가 된다.
4. 취약 상태가 된 넥서스는 피해를 받을 수 있다.
5. 넥서스 체력이 0이 되면 해당 넥서스를 소유한 팀이 패배한다.

이 규칙을 사용하는 이유는 다음과 같다.

- 두 라인 중 한 라인만 성공적으로 밀어도 게임이 전진한다.
- 2 vs 2 소규모 전장에서 과도한 라인 관리 부담을 줄일 수 있다.
- 플레이어가 "타워를 먼저 부수고 넥서스를 공격한다"는 목표를 쉽게 이해할 수 있다.
- 4주 프로토타입에서 승패 흐름을 명확하게 만들 수 있다.

## 구조물 구현 방향

구조물은 개별 타워 클래스를 먼저 만들기보다, 공통 기반인 `TowerBase`를 먼저 설계한다.

`TowerBase`의 목적은 이번 MVP의 방어타워뿐 아니라, 이후 다양한 구조물로 확장 가능한 공통 구조를 제공하는 것이다.

### TowerBase 공통 기능

`TowerBase`는 다음 공통 기능을 가진다.

- 팀 소유권
- 체력
- 피해 수신
- 파괴 처리
- 공격 가능 상태
- 타겟 판정용 충돌
- 서버 권한 피해 처리
- 파괴 이벤트 브로드캐스트
- UI 표시용 상태 제공

이번 4주 프로토타입에서 방어타워는 `TowerBase`를 상속하거나 조합해 만든다.

### 방어타워

방어타워는 이번 MVP의 핵심 구조물이다.

필수 기능:

- 팀별 소유권을 가진다.
- 적 플레이어의 공격으로 피해를 받는다.
- 체력이 0이 되면 파괴된다.
- 파괴 시 해당 팀 넥서스의 공격 가능 여부를 갱신한다.

이번 개발 기간에는 방어타워의 자동 공격 기능을 필수로 보지 않는다. 단, `TowerBase` 구조는 추후 공격 기능을 붙일 수 있도록 설계한다.

추후 확장 후보:

- 자동 타겟 탐색
- 적 플레이어 공격
- 적 미니언 공격
- 라인별 공격 우선순위
- 투사체 또는 히트스캔 공격
- 공격 범위 표시

### 넥서스

넥서스는 이번 4주 프로토타입에서 승패 목표 구조물로 사용한다.

필수 기능:

- 팀별 소유권을 가진다.
- 기본 상태에서는 피해를 받지 않는다.
- 아군 방어타워 중 하나라도 파괴되면 피해를 받을 수 있다.
- 체력이 0이 되면 매치 종료와 승패 판정을 발생시킨다.

이번 개발 기간에는 넥서스에 공격 기능을 넣지 않는다.

다만 넥서스도 구조적으로는 추후 공격 기능이나 특수 방어 기능을 붙일 수 있어야 한다. 따라서 넥서스를 단순한 체력 오브젝트로만 만들지 않고, `TowerBase`와 같은 구조물 공통 인터페이스 또는 기반 클래스를 공유하는 방향을 우선 검토한다.

추후 확장 후보:

- 넥서스 자동 방어 공격
- 넥서스 주변 보호막
- 타워 생존 여부에 따른 피해 감소
- 특정 조건에서만 피해 가능
- 넥서스 체력 단계별 효과

### 구현 주의점

구조물 구현 시 다음 기준을 지킨다.

- 승패 판정은 클라이언트가 아니라 서버 권한으로 처리한다.
- 넥서스 공격 가능 여부는 서버 상태를 기준으로 복제한다.
- 타워 파괴 이벤트가 넥서스 상태 변경으로 연결되어야 한다.
- 타워와 넥서스의 체력, 팀, 파괴 상태는 디버그 UI 또는 로그로 확인 가능해야 한다.
- 이번 MVP에서는 구조물 밸런스보다 상태 전이와 승패 연결을 우선 검증한다.

## 리스폰과 매치 시간

2주 MVP에서는 고정 리스폰과 시간 제한 없는 매치 구조를 사용한다.

기본 규칙은 다음과 같다.

- 플레이어가 사망하면 일정 시간 뒤 아군 스폰 지점에서 부활한다.
- 초기 리스폰 시간은 고정값으로 둔다.
- 사망 횟수나 매치 진행 시간에 따라 리스폰 시간이 증가하지 않는다.
- 매치 제한 시간은 두지 않는다.
- 승패는 넥서스 파괴로만 결정한다.

이 규칙을 사용하는 이유는 다음과 같다.

- 구현이 단순하다.
- 2주 MVP에서 필요한 승패 흐름이 명확하다.
- 사망 시간 증가나 제한 시간 승패 판정처럼 밸런스가 필요한 요소를 피할 수 있다.
- 플레이어가 계속 전장에 복귀하므로 테스트 반복 속도가 빠르다.

### 리스폰 구현 기준

리스폰은 서버 권한으로 처리한다.

필수 흐름:

1. 체력이 0이 된다.
2. 사망 상태로 전환한다.
3. 입력 또는 공격을 막는다.
4. 고정 시간 대기 후 팀 스폰 지점으로 이동한다.
5. 체력을 회복한다.
6. 다시 조작 가능 상태로 전환한다.

초기 리스폰 시간은 5초를 기준 후보로 둔다. 이 값은 밸런스 값이 아니라 테스트 흐름을 위한 임시값이다.

### 매치 종료

매치는 다음 조건에서만 종료된다.

- 한 팀의 넥서스 체력이 0이 된다.

2주 MVP에서는 다음 종료 조건을 넣지 않는다.

- 제한 시간 종료
- 킬 수 비교
- 타워 수 비교
- 넥서스 체력 비교
- 점수제 승리

제한 시간과 보조 승리 조건은 3~4주차 폴리싱 또는 시연 요구가 생길 때 추가 검토한다.

## UI, 로비, 매치메이킹, 채팅

4주차에 완성된 프로토타입처럼 보이기 위해 UI/HUD는 단순 디버그 표시를 넘어서 시연 가능한 수준을 목표로 한다.

다만 미니맵은 이번 4주 프로토타입의 우선순위에서 제외한다.

### 인게임 HUD

인게임 HUD의 필수 표시 항목은 다음과 같다.

- 플레이어 체력
- 기본 공격 상태
- 공격 스킬 쿨다운
- 이동 스킬 쿨다운
- 궁극기 쿨다운
- 현재 팀
- 사망/리스폰 상태
- 방어타워 상태
- 넥서스 상태
- 승패 메시지

HUD는 그래픽 완성도보다 정보 전달을 우선한다.

### TAB 리더보드

TAB 키를 누르면 리더보드를 볼 수 있어야 한다.

초기 리더보드 항목:

- 플레이어 이름 또는 임시 ID
- 팀
- 킬
- 데스
- 어시스트는 선택 사항
- 피해량은 선택 사항
- 현재 접속 상태

2주 MVP에서는 킬/데스와 팀 표시만 있어도 된다. 3~4주차에 어시스트, 피해량, 구조물 피해량 등을 확장 후보로 둔다.

### 로비

4주 프로토타입에는 간단한 로비 흐름이 필요하다.

목표:

- 플레이어가 매치에 들어가기 전 대기할 수 있다.
- 접속한 플레이어 목록을 확인할 수 있다.
- 팀 배정 상태를 확인할 수 있다.
- 준비 완료 또는 시작 버튼을 통해 매치를 시작할 수 있다.

2주 MVP에서는 완성형 로비보다, 서버 접속 후 매치 시작 전 최소 대기 화면 수준을 목표로 한다.

### 매치메이킹

이번 프로젝트에서 매치메이킹은 상용 서비스 수준의 자동 매칭이 아니라, 프로토타입 시연을 위한 단순 매치 생성/참가 흐름으로 정의한다.

필수 목표:

- 방 생성 또는 서버 접속
- 플레이어 참가
- 팀 배정
- 인원 조건 확인
- 매치 시작

제외 항목:

- 실력 기반 매칭
- 지역/핑 기반 매칭
- 랭크 매칭
- 큐 분리
- 매치 재검색
- 탈주 패널티

4주 프로토타입의 최종 목표는 2 vs 2 매치 시작이 자연스럽게 보이는 것이다.

### 실시간 채팅

실시간 채팅은 필요 기능에 포함한다.

채팅창은 단순 대화 기능뿐 아니라 킬로그와 시스템 알림을 통합 표시하는 피드로 사용한다.

채팅창에 표시할 항목:

- 플레이어 채팅
- 킬로그
- 사망 알림
- 리스폰 알림
- 타워 파괴 알림
- 넥서스 취약화 알림
- 넥서스 파괴 알림
- 승패 알림

초기 구현에서는 전체 채팅만 제공해도 된다. 팀 채팅은 3~4주차 확장 후보로 둔다.

### 미니맵 제외

미니맵은 이번 4주 프로토타입 우선순위에서 제외한다.

제외 이유:

- 2라인 소규모 맵에서는 HUD 알림과 채팅 피드만으로도 목표 전달이 가능하다.
- 미니맵은 구조물, 플레이어, 시야, 핑, 팀 정보 표시 등 부가 구현 비용이 크다.
- TPS 조작과 전투, 서버 권한 매치 흐름을 먼저 검증하는 것이 더 중요하다.

## UI 기능 우선순위

2주 MVP 필수:

- 플레이어 체력
- 스킬 쿨다운 4개
- 팀 표시
- 사망/리스폰 표시
- 타워/넥서스 상태 표시
- 승패 메시지
- 기본 접속/매치 시작 흐름
- 기본 로비
- 로비에서 팀 배정 상태 확인
- 로비에서 매치 시작

3~4주차 목표:

- TAB 리더보드
- 로비 화면 개선
- 단순 매치메이킹 흐름
- 실시간 채팅
- 채팅창 통합 킬로그/알림

제외:

- 미니맵
- 상용 수준 매치메이킹
- 랭크/큐 시스템
- 복잡한 점수판 통계

## 1~2주차와 3~4주차 경계

2주 MVP는 인게임만 따로 만드는 것이 아니라, 기본 로비에서 매치 시작까지 이어지는 최소 흐름을 포함한다.

2주차 종료 시점의 필수 흐름:

1. 서버 또는 방에 접속한다.
2. 로비 화면에 들어간다.
3. 접속한 플레이어 목록을 확인한다.
4. 팀 배정 상태를 확인한다.
5. 매치를 시작한다.
6. 팀별 스폰 지점에 배치된다.
7. 인게임 플레이가 진행된다.
8. 타워 파괴 후 넥서스를 공격한다.
9. 넥서스 파괴로 승패가 결정된다.
10. 승패 메시지가 표시된다.

2주 MVP에 포함하지 않는 항목:

- 실시간 채팅
- TAB 리더보드
- 킬로그
- 상세 시스템 알림 피드
- 로비 UI 폴리싱
- 매치메이킹 UX 폴리싱
- 재매칭
- 미니맵

3~4주차에는 위 항목 중 시연 완성도에 직접 영향을 주는 기능을 붙인다.

3~4주차 우선순위:

1. 실시간 채팅
2. 채팅창 통합 킬로그/시스템 알림
3. TAB 리더보드
4. 로비 화면 개선
5. 매치 시작/종료 UX 개선
6. 기본 사운드/피드백
7. 조작감과 전투 피드백 폴리싱

## 4주 주차별 산출물

### 1주차 산출물

1주차 목표는 `BaseCharacter`와 구조물 피해 흐름을 연결하는 것이다.

완료 기준:

- `BaseCharacter` 이동, 카메라, 조준 구현
- AbilitySystemComponent 연결
- 공통 AttributeSet 연결
- 기본 공격 Ability 구현
- 공격 스킬, 이동 스킬, 궁극기 Ability 슬롯 구조 구현
- Damage GameplayEffect 적용
- 체력 감소
- 사망 상태 전환
- 고정 리스폰 흐름 초안
- `TowerBase` 공통 구조 구현
- 방어타워 또는 테스트 구조물이 피해를 받음
- 구조물 체력이 0이 되면 파괴 상태로 전환됨
- 서버 권한으로 플레이어와 구조물 피해가 처리됨

1주차에 하지 않는 것:

- 완성형 로비
- 캐릭터 선택 UI
- 넥서스 승패 완성
- 실시간 채팅
- TAB 리더보드
- 미니언
- 정교한 밸런스

### 2주차 산출물

2주차 목표는 기본 로비에서 캐릭터를 선택하고, 매치를 시작해 넥서스 파괴로 승패가 나는 MVP를 완성하는 것이다.

완료 기준:

- 서버 또는 방 접속
- 기본 로비 화면
- 접속 플레이어 목록 표시
- 팀 배정 상태 표시
- Data Table 기반 캐릭터 목록 로딩
- 로비에서 캐릭터 선택
- 선택한 캐릭터 데이터로 스폰
- 매치 시작
- 팀별 스폰
- 방어타워 파괴
- 방어타워 중 하나가 파괴되면 넥서스 공격 가능 상태 전환
- 넥서스 피해 및 파괴
- 서버 권한 승패 판정
- 승패 메시지 표시

2주차의 캐릭터 선택은 캐릭터 수를 많이 늘리기 위한 기능이 아니라, 이후 파생 캐릭터를 빠르게 추가할 수 있는 데이터 기반 구조 검증이 목적이다.

### Data Table 기반 캐릭터 선택

캐릭터 종류와 선택은 Data Table을 기반으로 한다.

캐릭터 Data Table에는 다음 정보를 포함하는 것을 기준으로 한다.

- 캐릭터 ID
- 표시 이름
- 설명
- 역할군
- 사용할 Character Class 또는 Pawn Class
- 시작 Attribute 값
- 기본 공격 Ability
- 공격 스킬 Ability
- 이동 스킬 Ability
- 궁극기 Ability
- 기본 GameplayTag
- UI 표시용 아이콘 또는 임시 이미지 참조

2주차에는 `BaseCharacter` 기반의 선택 가능 캐릭터 2종을 Data Table에 등록한다.

두 캐릭터는 최종 캐릭터일 필요는 없다. 목적은 콘텐츠 수를 늘리는 것이 아니라, 같은 기반 구조에서 Attribute와 Ability 구성을 바꾸는 것만으로 다른 캐릭터를 빠르게 만들 수 있음을 검증하는 것이다.

캐릭터의 구체적인 역할, 스킬 방향, 원전 캐릭터는 추후 실제 투입 캐릭터 후보가 확정된 뒤 결정한다.

이 구조의 목적은 다음과 같다.

- 캐릭터 추가 시 코드 수정량을 줄인다.
- GAS Ability 조합을 데이터로 바꿀 수 있게 한다.
- 파생 캐릭터의 Attribute와 스킬 구성을 빠르게 실험한다.
- 4주 이후 5 vs 5 확장과 다수 캐릭터 제작에 대비한다.
- 이후 신규 캐릭터 추가 난이도를 크게 낮춘다.

### 3주차 산출물

3주차 목표는 시연 완성도를 높이는 온라인 기능과 정보 표시를 추가하는 것이다.

우선순위:

- 실시간 채팅
- 채팅창 통합 킬로그
- 채팅창 통합 시스템 알림
- TAB 리더보드
- 로비 화면 개선
- 매치 시작/종료 UX 개선
- 가능하면 파생 캐릭터 1종 추가

### 4주차 산출물

4주차 목표는 새 기능 확장보다 플레이 가능한 프로토타입처럼 보이게 다듬는 것이다.

우선순위:

- 조작감 개선
- 공격/피격 피드백
- 사망/리스폰 피드백
- 구조물 파괴 피드백
- 승패 연출
- 기본 사운드
- UI 가독성
- 네트워크 안정성
- 버그 수정
- 2 vs 2 시연 흐름 검증

4주차에는 새 시스템을 크게 추가하지 않는다. 미니언, 추가 캐릭터, 상세 통계 등은 일정이 안정적일 때만 선택한다.

## 팀 구성 가정

이번 4주 프로젝트는 프로그래머 5명 기준으로 진행한다.

다만 각 프로그래머의 숙련도와 전문 분야가 확실하지 않으므로, 전체 팀을 주니어급 프로그래머 5명으로 가정하고 업무를 분배한다.

이 가정에서 중요한 기준은 다음과 같다.

- 한 사람에게 네트워크, GAS, UI, 게임 규칙을 동시에 맡기지 않는다.
- 각 담당자는 명확한 산출물을 가진다.
- 기능 단위는 작게 나눈다.
- 1주차에는 구조 설계와 핵심 기술 검증을 우선한다.
- 2주차에는 MVP 완성을 위해 새 기능보다 연결과 마감에 집중한다.
- 3~4주차에는 각자 맡은 영역을 폴리싱하되, 위험한 신규 시스템 추가를 최소화한다.

### 5인 역할 분배 원칙

주니어급 팀 기준 역할은 다음처럼 나눈다.

| 역할 | 담당 영역 | 주의점 |
| --- | --- | --- |
| A | 프로젝트 리드 보조, GameMode/GameState, 매치 흐름 | 최종 판단자라기보다 통합 담당으로 둔다. |
| B | `BaseCharacter`, 이동, 카메라, 입력 | 전투 수치보다 조작과 상태 전이를 안정화한다. |
| C | GAS, Attribute, Ability, GameplayEffect | Ability 4종의 틀을 만들되 복잡한 효과는 피한다. |
| D | 구조물, `TowerBase`, 넥서스, 승패 판정 | 타워 파괴와 넥서스 취약화 연결을 책임진다. |
| E | 로비, 캐릭터 선택, HUD/UI | Data Table 기반 선택과 기본 HUD를 담당한다. |

실시간 채팅, TAB 리더보드, 킬로그/알림 통합은 3주차 이후 E가 주도하되, A가 GameState 데이터와 이벤트 연결을 지원한다.

### 주니어 팀 기준 리스크 관리

주니어급 5인 팀에서는 다음 항목을 특히 조심한다.

- 기능을 병렬로 많이 벌리기보다, 매주 동작하는 빌드를 만든다.
- GAS는 확장 가능성을 열되, 복잡한 상속과 추상화를 과도하게 만들지 않는다.
- 네트워크 처리는 서버 권한 기준을 명확히 하되, 예측과 보정은 MVP 범위에서 최소화한다.
- UI는 예쁘게 만들기보다 플레이 상태를 읽을 수 있게 한다.
- Data Table 구조는 캐릭터 확장을 돕는 수준으로 제한하고, 에디터 자동화나 복잡한 툴 제작은 하지 않는다.
- 4주차에는 새 시스템을 넣지 않는 것을 기본 원칙으로 한다.

## 현재 결정 요약

- 4주차 목표는 완성된 느낌의 프로토타입이다.
- MVP 자체는 2주 안에 완성한다.
- 장르 축은 TPS 팀 전투와 단순 AOS 목표의 절충형이다.
- 전장은 2라인 구조로 잡는다.
- 팀별 넥서스 1개와 1단계 방어타워를 둔다.
- 큰 맵은 우선순위가 낮다.
- 미니언은 2주 MVP 필수 범위에서 제외한다.
- 미니언은 3주차 이후 선택 확장으로 두며, 추가 시 근거리/원거리/힐러 3종을 기본 후보로 한다.
- 장기 목표 인원은 5 vs 5까지 확장 가능한 구조다.
- 이번 4주 프로토타입의 최종 목표 인원은 2 vs 2다.
- 2주 MVP의 안정성 기준은 최소 1 vs 1로도 한 판이 끝까지 돌아가는 것이다.
- 1주차에는 `BaseCharacter` 1종을 완성하고, 이후 상속 구조와 GAS를 통해 파생 캐릭터를 빠르게 제작할 수 있게 한다.
- 스킬 구조는 기본 공격, 공격 스킬, 이동 스킬, 궁극기 4개 입력을 기준으로 한다.
- 현재 단계에서 정교한 밸런스는 목표로 두지 않는다.
- 방어타워 중 하나라도 파괴되면 해당 팀 넥서스를 공격할 수 있다.
- 구조물은 `TowerBase` 공통 구조를 먼저 설계하고, 방어타워와 넥서스가 추후 공격 기능 등으로 확장 가능하게 만든다.
- 이번 4주 프로토타입에서 넥서스 공격 기능은 구현하지 않는다.
- 리스폰은 고정 시간 방식으로 처리한다.
- 매치 제한 시간은 두지 않고, 넥서스 파괴만 승패 조건으로 사용한다.
- UI/HUD는 시연 가능한 수준을 목표로 한다.
- TAB 리더보드, 로비, 단순 매치메이킹, 실시간 채팅을 4주 목표에 포함한다.
- 킬로그와 시스템 알림은 채팅창에 통합한다.
- 미니맵은 이번 4주 프로토타입에서 제외한다.
- 2주 MVP에는 기본 로비를 포함하고, 채팅과 리더보드는 3주차 이후 목표로 둔다.
- 1주차 종료 기준은 `BaseCharacter`와 구조물 피해 흐름 완성이다.
- 2주차 종료 기준은 기본 로비, Data Table 기반 캐릭터 선택, 넥서스 파괴 승패까지 포함한 MVP 완성이다.
- 2주차 캐릭터 선택은 `BaseCharacter` 기반 테스트 캐릭터 2종을 목표로 한다.
- 팀은 프로그래머 5명 기준이며, 전원 주니어급으로 가정해 보수적으로 업무를 분배한다.
